# 솔리가드(SoliGuard) 제품 기획서 & 개발 계획서

> 문서 버전 1.0 / 작성일 2026-06-06 / 대상: 솔리데오 내부 개발팀 및 의사결정자

---

## 1. 제품 개요

솔리가드는 SI 사업장에서 근무하는 실무자(개발자, 디자이너, 기획자, PM, 전산 사무직)의 PC에 설치되어, PC 안에 흩어진 개인정보와 민감정보를 스캔·검출하고, 마스킹·격리·완전삭제로 조치하며, 그 결과를 진단 리포트와 감사 로그로 남기는 데스크톱 보안 도구다.

제품의 한 줄 정의는 **"SI 수행사 실무자가 일상 업무 중 스스로 점검하는, 직무 맞춤형 경량 개인정보 자가점검 도구"**이다. 슬로건은 **"프로젝트가 끝나면, 데이터도 깨끗하게 — 솔리가드"**.

---

## 2. 배경 및 문제 정의

2025년 국내에서 신고된 개인정보 유출 사고의 원인 중 약 30%가 해킹이 아닌 "업무과실"이었다. SI 환경은 이 위험이 특히 크다. 개발자 PC에는 운영 DB에서 추출한 테스트 데이터와 실고객 정보 엑셀·로그가 쌓이고, 기획자·PM의 산출물(요건정의서, 테스트 시나리오)에는 실고객 샘플이 들어가며, 디자이너의 시안에는 신분증·계약서 스캔본이 포함된다. SI 사업은 발주처(고객사)의 데이터를 다루기 때문에 유출 시 회사가 계약 위반과 법적 책임을 함께 진다.

기존 시장은 양극화돼 있다. 셜록홈즈·U-PRIVACY SAFER 같은 기업용 솔루션은 관리서버+에이전트 구조로 무겁고 고가이며 발주처(공공·대기업)가 주로 쓴다. 인포스캔류 개인 무료 도구는 가볍지만 정규식 단순 매칭으로 오탐이 많고 노후화됐으며 SI 산출물(소스코드·로그·API키)을 다루지 못한다. 그 사이, "SI 수행사 실무자가 직무에 맞게 스스로 점검하는 경량·정확 도구"라는 자리가 비어 있다.

---

## 3. 타겟 사용자 및 핵심 시나리오

주 사용자는 SI 사업장의 개발자, 디자이너, 기획자, PM, 전산 사무직이다. 부차적으로는 이들의 점검 이행을 관리하는 팀 리더/보안 담당자다.

핵심 시나리오는 다음과 같다.

- **개발자**: 프로젝트 작업 폴더와 소스 디렉터리를 스캔해 하드코딩된 DB 접속정보·API 키, 운영 데이터 추출 파일을 찾아 정리한다.
- **기획자·PM**: 산출물 폴더를 스캔해 문서 속 실고객 샘플을 마스킹한다.
- **디자이너**: 시안·리소스 폴더를 스캔해 신분증·계약서 스캔 이미지를 OCR로 검출해 격리한다.
- **공통(프로젝트 클로징 점검)**: 프로젝트 종료 시 잔여 발주처 데이터를 일괄 검출·삭제하고 진단서를 발급해 보안 증빙으로 남긴다.

---

## 4. 핵심 기능 (기능 명세)

제품의 동작은 **스캔 → 검출 → 분류 → 조치 → 기록**의 5단계로 구성된다.

- **스캔**: 사용자가 지정한 폴더 또는 드라이브 전체를 대상으로 파일을 순회하며 내용을 검사한다. 설치 시 선택한 직무(개발자/디자이너/기획자/PM/사무)에 따라 우선 스캔할 폴더와 파일 확장자를 자동 구성하는 **역할별 스캔 프로파일**을 제공한다.
- **검출**: 2단계로 수행한다. 1단계는 정규식으로 후보를 빠르게 찾고, 2단계는 검증 알고리즘으로 진짜 여부를 가린다. 주민등록번호는 생년월일·성별코드 유효성과 체크섬을 계산해 검증하고, 신용카드번호는 Luhn 알고리즘으로, 사업자등록번호는 검증 공식으로 확인한다. 이 2단계 검증이 무료 도구 대비 오탐을 줄이는 핵심 차별점이다.
- **분류**: 검출 결과를 정보 유형과 위험도(높음/중간/낮음)로 나눠 목록화하고, 파일 경로·검출 항목·개수를 보여준다.
- **조치**: 세 가지 옵션을 제공한다.
  - *마스킹*: 검출 부분을 별표로 가린 사본을 만들거나 원본을 변환한다.
  - *격리*: 위험 파일을 암호화해 별도 격리 폴더로 옮기고 복원 가능하게 보관한다.
  - *완전삭제*: 복구 불가능한 덮어쓰기(wiping)로 영구 삭제하며, 실수 방지를 위해 반드시 격리를 거친 뒤 확인 단계를 둔다.
- **기록**: 모든 스캔·조치 내역을 감사 로그로 남기고, 점검 결과를 PDF 진단 리포트로 발급한다. 이 리포트와 로그는 개인정보보호법 대응 및 발주처 보안 감사 증빙으로 활용된다.

### 검출 대상별 처리 방식

| 정보 유형 | 1차 탐지 | 2차 검증 | 대상 직무 |
|---|---|---|---|
| 주민등록번호 | 정규식 패턴 | 생년월일·성별·체크섬 | 전체 |
| 신용카드번호 | 13~16자리 패턴 | Luhn 알고리즘 | 전체 |
| 사업자등록번호 | `\d{3}-\d{2}-\d{5}` | 검증 공식 | 전체 |
| 휴대전화/전화번호 | 정규식 | 번호대 확인 | 전체 |
| 이메일·여권·운전면허 | 정규식 | 형식 검증 | 전체 |
| 계좌번호 | 은행별 자릿수 | 길이·구분자 | 전체 |
| DB 접속정보·API 키·하드코딩 비밀번호 | 키워드+패턴 | 엔트로피 검사 | 개발자 |
| 이미지 속 정보(신분증·계약서) | OCR 후 위 로직 | — | 디자이너 |
| 문서 내 실고객 샘플 | 정규식+문맥 | — | 기획자·PM |

지원 파일 포맷은 txt, csv, xlsx, xls, docx, hwp, hwpx, pdf, 소스코드 파일(다양한 확장자), 그리고 OCR을 통한 jpg, png, bmp까지 포함한다. 한글(hwp/hwpx) 파싱은 국내 SI 환경의 필수 차별점이다.

## 5. 차별화 전략 요약

솔리가드의 차별점은 네 가지다.

1. **직무별 스캔 프로파일**로 개발자·디자이너·기획자·PM 각자에게 맞는 점검을 제공한다.
2. **소스코드·로그의 DB 접속정보·API 키·비밀번호 검출**로 일반 개인정보 도구가 못 하는 SI 산출물을 다룬다.
3. **프로젝트 클로징 점검**으로 프로젝트 종료 시 잔여 데이터를 일괄 정리한다.
4. **100% 로컬 처리**로 검출된 데이터를 외부로 전송하지 않아 발주처 데이터 보안을 보장하며, 이는 발주처와의 보안 계약(SLA) 준수에 직결된다.

## 6. 기술 아키텍처 설계

전체 구조는 단독 실행형 데스크톱 앱을 기본으로 하고, 옵션으로 경량 팀 대시보드를 둔다. 무거운 서버-에이전트 구조는 채택하지 않는다.

구성 요소는 네 개의 계층으로 나눈다.

- **UI 계층**: 사용자가 스캔을 실행하고 결과를 확인·조치하는 화면을 담당한다.
- **검출 엔진 계층**: 파일 파서, 정규식 매처, 검증 모듈(체크섬·Luhn·엔트로피), OCR 모듈로 구성되며 제품의 심장이다.
- **조치 계층**: 마스킹·암호화 격리·완전삭제를 수행한다.
- **기록 계층**: 로컬 DB에 감사 로그를 저장하고 PDF 리포트를 생성한다.

### 권장 기술 스택

- **핵심 언어**: 검출 엔진과 OCR 연동에 강점이 있는 Python을 기본으로 한다.
- **GUI**: 배포와 네이티브 성능을 고려해 PySide(Qt) 또는 Electron 중 택일하며, 개발 리소스가 Python 중심이면 PySide를 권한다.
- **OCR**: 한국어 학습 데이터를 갖춘 Tesseract를 1차로 쓰되, 정확도가 부족하면 상용 OCR API(네이버 클로바 등) 연동 옵션을 둔다. 단 OCR API 사용 시에도 이미지가 외부로 나가므로, 기본값은 로컬 OCR로 두고 사용자가 명시적으로 동의한 경우에만 외부 OCR을 쓰도록 설계한다.
- **파싱**: 한글 파일 파싱, PDF 파싱, Office 파싱은 각각의 라이브러리를 활용한다.
- **로컬 DB**: SQLite.
- **격리 암호화**: 국정원 인증 알고리즘 또는 표준 강암호(AES-256)를 적용한다.

검출 엔진은 정규식과 검증 함수를 독립 모듈로 설계해, 추후 문맥 기반 AI 판단(숫자가 실제 개인정보 맥락에 쓰였는지 판단)으로 고도화할 수 있게 한다.

## 7. 개발 로드맵 (단계별 계획)

개발은 세 단계로 나눠 진행한다. 아래 기간은 소규모 팀(개발 2~3명, 기획/디자인 각 1명) 기준의 가늠치이며, 실제 인력 구성에 따라 조정한다.

### 1단계 — MVP (약 8~10주)

제품의 심장인 검출 엔진과 기본 흐름을 완성한다. 텍스트 기반 파일(txt, csv, docx, xlsx, hwp, pdf)에 대한 2단계 검출(주민번호·카드·사업자번호·전화·이메일·계좌), 기본 GUI(폴더 지정 스캔 → 결과 목록 → 마스킹/격리/삭제 조치), 감사 로그와 SQLite 저장, 그리고 개발자·기획자용 기본 스캔 프로파일을 구현한다. 1단계 종료 시점에 사내 일부 인원으로 파일럿 테스트를 한다.

### 2단계 — 차별화 기능 (약 8주)

SI 특화 기능을 더한다. 소스코드·로그 내 DB 접속정보·API 키·비밀번호 검출(엔트로피 검사 포함), OCR 기반 이미지 속 신분증·계약서 검출, 직무별 스캔 프로파일 전체(디자이너·PM 포함), 프로젝트 클로징 점검, PDF 진단 리포트 발급을 구현한다.

### 3단계 — 운영·확장 (약 6주)

정기 자동 스캔(스케줄링), 경량 팀 대시보드(구성원 점검 이행 현황 요약 공유), 위험도 점수·요약 지표를 추가하고, 사내 전사 배포를 위한 설치 패키지·업데이트 체계를 정비한다.

전체적으로 약 5~6개월 안에 사내 배포 가능한 수준을 목표로 한다.

## 9. 리스크 및 대응

1. **검출 정확도**: 오탐이 많으면 사용자가 도구를 신뢰하지 않고 떠난다. 2단계 검증과 화이트리스트(검증된 더미 패턴 제외)로 관리하고, 파일럿에서 정탐/오탐 비율을 측정해 임계값을 조정한다.
2. **OCR 정확도와 데이터 외부 유출 우려**: 로컬 OCR을 기본값으로 두고 외부 API는 명시적 동의 시에만 쓰도록 설계해 신뢰 문제를 차단한다.
3. **한글(hwp)·다양한 파일 포맷 파싱의 안정성**: 포맷별 파싱은 깨진 파일·암호 걸린 파일에서 오류가 잦으므로, 파싱 실패 시 해당 파일을 "검사 불가"로 분리 기록하고 전체 스캔이 멈추지 않도록 예외 처리를 견고히 한다.
4. **완전삭제의 비가역성**: 사용자가 실수로 중요 파일을 영구삭제하지 않도록, 삭제는 반드시 격리 단계를 거치고 명시적 확인을 받도록 강제한다.

## 10. 성공 지표(KPI)

도구의 성공은 다음으로 측정한다.

- 사내 채택률(설치·정기 점검 실행 비율)
- 검출 정확도(오탐률 목표치 설정)
- 프로젝트 클로징 점검 이행률
- 검출·조치된 위험 파일 건수

사내 안착 후에는 외부 SI 기업 대상 제품화 가능성을 검토한다.
